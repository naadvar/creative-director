# Ideas from your DNA — feature spec (MVP shipped 2026-07-01)

_Grounded reel ideation: 'what should I make next?' answered from the creator's
OWN craft profile — never trends. Spec from a 3-lens design panel; the bar:
if a generic LLM with no context could produce the idea, it must not ship._

## Recommendation

Ship "Your next reel" as a section inside the existing Growth page (MyDnaPage.tsx): ONE idea at a time, generated by a single text-only LLM call on the existing DeepInfra Qwen rails (settings.craft_read_base_url), grounded in three server-assembled inputs — the creator's last 5 grounded Upload.craft_read rows (with video_ids), the deterministic fingerprint/progress recurring-gap stats, and a memoized niche craft digest computed from the 6.8k-reel corpus (real percentages ChatGPT cannot know). Honesty is enforced the same way the rest of the product enforces it: all numbers are server-stamped (the model is forbidden from writing statistics), every idea must cite real video_ids from the creator's own uploads (deterministically validated, fabricated citations are rejected), and failed validation after one retry yields an honest empty state instead of filler. The flywheel closes with the cheapest possible link — a "Plan this next" button that carries idea_id into the existing upload form (same pattern as the revision loop's ?prior= param) and stamps Upload.idea_id, so "did the planned guardrail hold" analysis is unlockable later without any MVP verifier. Cache invalidation is automatic via a cache_key derived from the creator's grounded reads, so a new read always produces a fresh idea — the retention hook.

## Prompt design

MODULE: creative_director/profile/ideas.py (new). One text-only call to settings.craft_read_base_url (same httpx POST pattern as _ground_post in creative_director/advice/craft_xray.py: response_format json_object, Authorization bearer settings.craft_read_api_key, model settings.craft_read_model), but temperature 0.9 (NOT the 0 used for reads — ideation needs variety across regenerates; validation makes this safe), max_tokens 900, timeout 120.

SYSTEM PROMPT (verbatim structure):
"You are the same craft critic who wrote this creator's reel reads — now planning their NEXT reel. Produce ONE concrete, shootable reel concept grounded ONLY in (a) the creator's own reels and craft notes and (b) aggregate craft patterns for their niche, both given below.
HARD RULES:
- CITE: grounded_in and strength_used must reference video_ids from the CREATOR DNA list. Every video_id you write MUST appear in that list — never invent a reel.
- CRAFT, NEVER PERFORMANCE: never mention views, reach, followers, engagement, virality, trends, trending audio, challenges, or the algorithm. The idea is judged as craft — clearer, tighter, more watchable — never as performance.
- NO NUMBERS: write no statistics, counts, or percentages anywhere; the app attaches real ones itself.
- PRE-EMPT THE GAP: structure the beat_sheet so the creator's TOP CRAFT GAP cannot recur — plan the fix into the shoot itself (e.g. gap text_illegible → specify where the text sits, how large, how long it holds, before they shoot).
- NEW, NOT A REMIX: the concept must not restate any reel in the CREATOR DNA list. Build on a strength; don't repeat a video.
- SHOOTABLE: phone-camera realistic, one or two sessions, no crew.
- NICHE-BOUND: if the concept would read as a fine suggestion for any creator in any niche, it is wrong. It must only make sense for THIS creator's niche, format, and strengths.
Respond with ONLY one JSON object (no markdown fences, no prose) with EXACTLY these keys: "concept" (string, <=10 words), "premise" (string, 1-2 sentences), "format" (one of "talking_head","visual_led","mixed"), "grounded_in" (array of 1-2 objects {"video_id","why"}), "strength_used" (object {"video_id","strength","how"}), "beat_sheet" (array of 3-5 objects {"beat","time","direction"}, times as m:ss ranges), "gap_guardrail" (object {"gap": exactly the gap key given below, "plan": string}), "shoot_notes" (string, one line). Strictly valid JSON."

USER CONTENT (assembled deterministically per request):
1. CREATOR DNA block — from Upload rows WHERE user_id=X AND craft_read IS NOT NULL AND grounded is not False, newest 5: one line each: "up_9f2ab31c77d0 (2026-06-21, visual_led): <what_it_is, 160 chars> | done_well: [...] | notes: [text_illegible, weak_framing]". Then "Dominant format: visual_led (4 of 5). Niche: ig_fitness." (from compute_fingerprint).
2. TOP CRAFT GAP TO PRE-EMPT — tiered by read count (see gating): at 3+ reads with fingerprint.recurring non-empty: "text_illegible — 'on-screen text that's hard to read at reel size' (recurred across their reads)"; at 1-2 reads: "in their last read the note was X — treat it as a watch-item, not a pattern". Reuse _CHANGE_FRIENDLY labels from fingerprint.py. Fall back to progress.recurring[0].dimension when change_types are empty.
3. STRENGTHS TO LEAN ON — top 1-2 done_well strings with their source video_ids.
4. NICHE CRAFT DIGEST — memoized (functools.lru_cache; corpus DB is read-only per deploy) SQL over video_features JOIN videos JOIN channels WHERE niche=? AND craft_read IS NOT NULL, parsed with json.loads, grounded-only: (a) "most common craft notes in ig_fitness (share of 4,769 grounded reels): text_illegible 37%, weak_framing 25%, hook_unclear 14%"; (b) 10 one-line what_it_is structure exemplars for the creator's dominant format, sampled from CLEAN reels (empty blind_spots), each truncated to 160 chars, prefixed "background knowledge — never name, quote, or reference any specific reel from this digest in your output".
5. TASK line: "Give ONE idea for their next reel. JSON only."

VALIDATION (deterministic, in-process, no LLM): parse via _loads_robust (import from craft_xray). Reject if: (a) any required key missing / format not in enum / beat_sheet not 3-5 items; (b) any video_id token (regex up_[0-9a-f]{12}) in the output is not in the creator's grounded set, or grounded_in is empty; (c) banned-phrase regex hits concept+premise+beats+guardrail: \b(viral|virality|trending|trend audio|trending audio|algorithm|views|reach|followers|engagement|blow up|go viral|day in the life|top \d+|you won.?t believe|challenge)\b (case-insensitive); (d) any digit followed by % or the words "percent|of reels|most creators" appears (model-authored stats forbidden); (e) difflib.SequenceMatcher ratio between premise and any of the creator's what_it_is summaries > 0.6 (remix rejection — stdlib, no embedding infra needed on the lean host); (f) gap_guardrail.gap != the served gap key. On failure: ONE regeneration with the failure reason appended to the user content ("Your previous idea was rejected because: ..."). On second failure: return {ready: false, reason} and log the suppression — honest silence over filler, same ethos as the grounding gate.

## Anti-generic mechanisms

- Citation-grounded generation: every idea must cite >=1 real video_id from the creator's grounded uploads; the prompt lists the only legal ids and a deterministic validator (regex-extract up_[0-9a-f]{12}, exact-match against the set) rejects any fabricated citation — a hallucinated reel cannot ship.
- Server-stamped numbers only: the model is forbidden from writing any statistic; gap_stat_line ('came up in 3 of your 5 reads') and digest_line ('37% of 4,769 fitness reels') are composed deterministically from fingerprint/progress and the corpus digest — the two most personalized lines on screen carry zero fabrication surface, same design as fingerprint.py's no-LLM aggregate.
- Niche craft digest injection: real corpus aggregates (verified live: ig_fitness grounded n=4,769 — text_illegible 37%, weak_framing 25%, hook_unclear 14%; per-format splits) plus 10 clean-reel what_it_is structure exemplars for the creator's dominant format go INTO the prompt — context no generic LLM session has.
- Pre-empted-gap requirement: gap_guardrail.gap must echo the exact served gap key (validated) and the beat_sheet must plan the fix into the shoot; the gap comes from their actual recurring change_types/opportunity_dimension, so the plan is structurally personal.
- Remix rejection: difflib.SequenceMatcher premise-vs-their-own-what_it_is ratio > 0.6 rejects restatements of reels they already made (stdlib, works on the lean host with no embedding stack).
- Banned trend-speak/performance regex over all model text: viral|trending|algorithm|views|reach|followers|engagement|day in the life|top N|challenge etc. — rejects commodity phrasing at the validator, not just in the prompt.
- Niche-bound instruction + one-idea-per-view: the prompt's 'if it would fit any creator in any niche, it is wrong' rule, and the UI shows exactly one deeply-grounded idea (a list of 5 is structurally slop).
- Suppression over filler: one retry with the rejection reason, then an honest {ready:false} empty state — the same say-nothing-rather-than-invent contract as the grounding gate, extended to ideation.

## Gating

Count = GROUNDED reads only (Upload.craft_read not null AND grounded is not False — same filter as compute_fingerprint). 0 reads: {ready:false, reason} → Growth shows 'Read a reel and ideas start here' CTA to /analyze; no generation call ever fires. 1 read: feature IS available (the retention moment when a new user decides the app's value; the single read + niche digest already make it non-generic) but language is tiered — caveat field set to 'Based on your first read — patterns firm up as you read more', gap served as 'in your last read the note was X — treat it as a watch-item, not a pattern', and the words 'recurring/always/tend to' are excluded from the server-composed gap_stat_line. 2 reads: same caveated tier unless fingerprint.recurring is non-empty (it only returns entries with count>=2, reusing fingerprint.py's existing honesty threshold) — if a note genuinely hit twice, recurring language unlocks: 'X came up in both of your reads.' 3+ reads: full tier — gap from fingerprint.recurring[0] (fallback progress.recurring[0]), stat line with real counts, digest comparison line included. BANNED CLAIMS (never in model output — validator-enforced; never in server copy — code-review-enforced): views/reach/followers/engagement/virality/trends/algorithm or any performance implication; any model-authored number or percentage; 'you always/never'; corpus facts not computed by the digest; 'this will work/perform'. REFRESH + CACHE: cache_key = sha1(user_id + sorted grounded video_ids + max(craft_read updated timestamp)) stored on the CreatorIdea row; GET /me/idea returns the newest row whose cache_key matches current state, else generates — so a new grounded read AUTO-invalidates (no delete hooks in upload.py needed) and a suppressed read changes nothing. ?fresh=1 ('Show me another') regenerates within the same cache_key, capped at 5 generations/user/day (count CreatorIdea rows by created_at; over cap → 429 with friendly copy). Rows are append-only (audit trail + future saved-ideas UI for free). Suppressed generations log user_id, n_reads, attempt count, rejection reasons via loguru.

## UI

Primary surface: a 'Your next reel' section on frontend/src/pages/MyDnaPage.tsx, directly BELOW the 'Your craft trend' card — Growth is where the DNA already lives, so 'ideas from your DNA' renders next to the fingerprint that explains them. NOT a new tab: the native IA is locked at New/Library/Growth (TestFlight Build 6 in flight) and a section inside Growth ships in the web bundle with zero iOS resubmission churn. The IdeaCard (new component) renders: concept + premise; 'Why this is yours' block = grounded_in/strength_used with tappable citation thumbnails linking to /video/{id} (the creator can verify the grounding in one tap — the trust affordance); numbered beat sheet; the gap guardrail as a highlighted callout with the server-stamped gap_stat_line + digest_line beneath it in small print; buttons: [Plan this next] → /analyze?idea={idea_id}, [Show me another] → ?fresh=1 (disabled with honest copy at the daily cap), and one-tap 'Helpful / Not for me' feedback (mirrors the NoteFeedback pattern). Empty/gated states per the gating spec (0 reads → CTA card; caveat banner at 1-2 reads; generation failure → 'No grounded idea this time — read another reel and this refreshes', never a fake idea and never 'error occurred'). Secondary entry: VideoPage.tsx already links to /my-dna (lines 154/204) — change that link copy on the creator's OWN grounded uploads to 'See your DNA + your next idea'. Loading state: 'Sketching your next reel…' spinner (generation is one text call, ~2-4s).

## Flywheel

Explicit linking, no fuzzy matching — the exact lesson from the revision loop (a heuristic 'did they shoot it?' matcher would fabricate closure the way the single-version verifier fabricated 'resolved'). [Plan this next] navigates to /analyze?idea={idea_id}; UploadPage.tsx reads the param exactly like the existing ?prior= revision pattern, shows a banner ('Shooting your DNA idea — {concept}'), and passes idea_id through api.upload() as a form field; upload.py stamps it onto the durable Upload row (new nullable uploads.idea_id column via the existing _USERDATA_RUNTIME_COLUMNS migration). That one column captures the idea→shoot→upload→read join from day one, so the phase-2 payoff — 'did the planned guardrail hold?' = compare the new read's change_types/opportunity_dimension against CreatorIdea.gap_dimension — is a pure offline query later, with zero MVP verifier work. The loop's cadence is driven by cache invalidation: every new grounded read changes the cache_key, so Growth always has a FRESH idea reflecting the latest read → prime to shoot → upload → new read → new idea. Feedback ('helpful'/'not_for_me' on the CreatorIdea row) is the cheapest quality signal for a brand-new generative surface and reuses the one-tap pattern the product already trained users on; it is logged for prompt tuning, not used for personalization in MVP.

## Deferred

- Phase-2 guardrail verification: after a linked upload's read completes, compare its change_types/opportunity_dimension to CreatorIdea.gap_dimension and surface 'the plan held / the note came back' — the Upload.idea_id join makes this a later offline query + small UI addition; do NOT ship any MVP claim about it.
- Saved-ideas shelf / idea history UI (rows are already append-only in CreatorIdea; purely frontend later).
- Effectiveness analytics script (idea → next-read gap-resolution rates per gap type, for prompt tuning) — needs weeks of linked data first.
- Per-reel ideation entry ('ideas inspired by this reel' on Library cards with a focus_video_id param).
- LLM-judge second validation pass — only if the measured slop/suppression rates from the deterministic validator prove insufficient; do not add speculatively (the grounding-gate lesson: don't re-oscillate on gate prompts).
- Push/email 'new idea ready' nudges; iOS-native surfacing changes or a fourth tab.
- Feedback-driven personalization (bias future ideas by helpful/not_for_me votes).
- Multi-niche creators (MVP uses fingerprint's dominant niche only).
- Idea variations / compare-two-ideas UI.

## Do NOT do

- No trend inputs of any kind — no trend feeds, no 'trending audio' suggestions, no seasonal/holiday hooks; the differentiator is the creator's own craft profile.
- Never expose corpus content in the app: no third-party reel ids, handles, titles, quotes, or thumbnails in the idea or UI (App Store IP cleanliness). Corpus exemplars go into the PROMPT only, with an explicit instruction never to reference them; only aggregate digest lines (counts/percentages) reach the screen.
- No performance claims anywhere — model output (validator-enforced) or UI copy: no views/reach/followers/engagement/virality/algorithm, no 'this will work'.
- No model-authored numbers: every count and percentage on screen is computed server-side from fingerprint/progress/digest.
- No idea lists: one idea per view, ever. A gallery of 5 is the slop failure mode by construction.
- No LLM validation pass in MVP — deterministic checks only; and don't iterate the generation prompt reactively without the live QA script (the grounding-gate lesson: validate via script, don't re-oscillate).
- No fuzzy 'did they shoot this idea?' inference — explicit idea_id linking only (the revision loop proved heuristic closure fabricates).
- No generation at 0 grounded reads, and no generic fallback idea when validation fails twice — honest empty state instead.
- No new persistent tables in the corpus DB — CreatorIdea and uploads.idea_id live in userdata.db or they get wiped on the next corpus redeploy.
- No embedding-similarity infrastructure for dedup (MiniLM isn't guaranteed on the lean serve host) — difflib is sufficient for MVP.
- No new native tab / iOS IA change and no push/email nudges in this release.
- Don't let 'Show me another' be unlimited — cap generations (cost is small, but unbounded regeneration trains users to slot-machine the feature, which cheapens each idea).

## Implementation notes (as built)
- Engine: creative_director/profile/ideas.py (DNA assembly, corpus digest, prompt, validator, cache, cap).
- Storage: CreatorIdea table + Upload.idea_id (userdata.db, boot-migrated).
- API: GET /me/idea (?fresh=1), POST /me/idea/{id}/feedback; upload accepts idea_id Form.
- UI: IdeaCard on Growth (MyDnaPage) + 'Shooting your DNA idea' banner on UploadPage (?idea=).
- Validator note: the model-stats ban is scoped to POPULATION claims ('37% of reels');
  spatial craft directions ('text at 80% width') are allowed — the strict version
  suppressed 3/3 live generations before this was narrowed.
- Live QA: 3/3 real-LLM generations validated on corpus-seeded DNA; regenerates get a
