# Revision / improvement loop — design (pending build)

_Designed via the revision-improvement design panel. Not yet built. MVP ≈ 1 day._

## Recommendation
Link revisions EXPLICITLY (creator taps "Re-check this fix" on a read, which carries prior_video_id into the next upload) — never auto-detect by CLIP/caption fingerprint, because a false link produces a fabricated "you fixed it," which is the one failure mode that violates the product's honesty core. Ground the verdict PER PRIOR BLIND_SPOT with a fresh, frames-only VLM re-check (a new verify_fix_addressed call in craft_xray.py that re-watches the NEW reel's frames at the prior issue's timestamp and asks only "is THIS still present" — it never sees the new read's text, so noisy run-to-run read variance cannot drive the verdict). Mark "Fixed" only when BOTH the prior change_type is absent from the fresh new read AND the frames-grounded verifier returns resolved with high confidence; anything weaker degrades to "Looks better / Still there / Can't tell," never silence-as-success. Persist a revision_verdict JSON on the new Upload so it survives corpus redeploys, and snapshot the prior issue's text+timestamp+change_type at link time so the verdict never depends on the (wipeable) old VideoFeatures row.

## Linking
EXPLICIT, creator-initiated linking — reject the auto-detect fingerprint design (#2). Mechanism: add nullable Upload.prior_video_id (String 64, indexed). On a read page (VideoPage.tsx), each blind_spot and the lever get a "Re-check this fix" CTA that routes to /analyze?prior=<video_id>. UploadPage reads the prior query param, shows a "Re-checking your fix to [prior title]" banner, and passes prior_video_id as an optional Form field to POST /upload. _run_job stamps it onto the new Upload row. Rationale grounded in code: the reads are explicitly non-deterministic (upload.py MAX_READ_ATTEMPTS retry exists precisely because a re-upload of the SAME reel often yields a different read), so visual/caption similarity (design #2's CLIP+caption fingerprint) cannot distinguish "improved revision" from "same reel re-run" — and a wrong link auto-emits a fabricated fixed claim. Explicit linking makes the creator assert intent, costs one tap, and the durable prior_video_id survives corpus redeploys (it lives in userdata.db Upload, unlike the wiped corpus rows). Snapshot prior issues at link time (see data_model) so the verdict never reads the old, possibly-wiped VideoFeatures.craft_read.

## Grounding the verdict (the crux)
The "Fixed" verdict is grounded by a fresh frames-only VLM re-check, never by diffing two read texts. Concretely, per prior blind_spot we require TWO independent gates to BOTH pass before claiming fixed: (1) ABSENCE gate — the prior blind_spot's change_type tag must NOT appear in the freshly-generated new read's change_types (cheap, reuses existing closed vocab from craft_xray.py CHANGE_TYPES); (2) PRESENCE gate — a new verify_fix_addressed(new_mp4, prior_issue_text, prior_timestamp, prior_change_type) call samples hi-res frames from the NEW mp4 (reusing sample_frames_hires) and asks the VLM ONLY: "A prior version of this reel had this specific craft issue at [ts]: [text]. Watching THESE frames, is that specific issue resolved, still present, or can't-tell? Ground only in what you SEE; invent nothing." It returns {status: resolved|still_there|cant_tell, confidence: high|medium|low, evidence}. Critically the verifier never sees the new read's text — so non-deterministic read variance (the core risk) cannot leak into the verdict; the verdict is anchored to the prior issue (stable, snapshotted) and fresh pixels. Fixed is emitted ONLY when gate 1 passes AND gate 2 returns resolved+high. Mapping of the four other cases: tag absent but verifier still_there/partial -> "Still there (with fresh, more specific evidence)"; tag absent + verifier resolved+medium -> "Looks better — check it yourself" (surfaces uncertainty, never asserts); verifier cant_tell or frames/endpoint unavailable -> "Can't verify automatically" (suppress that line, never a false success); new read itself suppressed by the grounding gate -> revision_verdict suppressed entirely, fall back to plain read. This mirrors the codebase's existing two-pass, confirmation-required suppression ethos (ground_and_gate requires gate+confirm agreement) and reuses extract_craft_lever's proven "re-watch the real frames" primitive rather than trusting a text summary (synthesize_opportunity is documented to confabulate ~15% from text alone).

## Verdict states (creator-facing)

- FIXED (green ✓) — gate1 absent AND verifier resolved+high. Copy: 'You fixed this. Last time: "[prior issue]" at [ts]. Re-watching your new frames, that's resolved. Tap [ts] to compare.' Only state that asserts success.
- LOOKS BETTER — CHECK IT YOURSELF (neutral accent) — verifier resolved+medium, OR gate1 absent but verifier confidence not high. Copy: 'This looks improved — the model no longer flags "[prior issue]", but I'm not fully certain it's the same moment. Tap [ts] and judge for yourself.' Surfaces uncertainty; never claims the fix.
- STILL THERE (amber) — verifier still_there (regardless of whether the tag dropped from the noisy new read). Copy: 'Still worth another pass. "[prior issue]" still shows at [ts] — re-watching, here's specifically what I see: [fresh evidence]. The original fix still applies: [prior Fix text].' Acknowledges the attempt, gives a MORE specific reason, never a generic repeat.
- CAN'T VERIFY (muted) — verifier cant_tell, or frames/endpoint/prior unavailable. Copy: 'I couldn't reliably re-check "[prior issue]" on this version — so I won't guess. Here's your fresh read below.' Suppress rather than fabricate.
- NEW SINCE LAST TIME (standard read styling, not framed as failure) — new read blind_spots whose change_type wasn't in the prior read. Copy: 'New this version: [observation]. Try: [fix].' Honest surfacing of newly-introduced or newly-visible issues without implying regression.

## Data model

- **Upload.prior_video_id** — String(64), nullable, indexed. The video_id of the reel this upload is a revision of. Set only on explicit creator re-check. Durable in userdata.db; survives corpus redeploys.
- **Upload.revision_verdict (JSON, nullable)** — {prior_video_id, prior_title, checked_at(ISO8601), items: [{prior_issue_text, prior_timestamp, prior_change_type, status: 'fixed'|'looks_better'|'still_there'|'cant_tell', confidence: 'high'|'medium'|'low', evidence, absence_gate: bool, verifier_status}], new_issues: [blind_spots in the new read whose change_type was NOT in the prior read], summary: {n_fixed, n_looks_better, n_still, n_unsure}}. Computed once at upload time, persisted (NOT recomputed on every page view) so it's stable and cheap to render.
- **PRIOR-ISSUE SNAPSHOT (no new table)** — The verdict embeds the prior issue text+timestamp+change_type copied from the prior Upload.craft_read at compute time. This is the load-bearing durability decision: the verdict must not depend on re-reading the prior read later (the prior reel could be deleted via DELETE /me/uploads, or its corpus VideoFeatures row wiped on redeploy). Snapshotting makes the verdict self-contained.
- **api response: craft_read endpoint** — {available, read, meta:{...,prior_video_id,prior_title}, revision_verdict}. No change to the craft_read JSON schema itself (SCHEMA_VERSION untouched) — revision_verdict is a sibling, so existing reads/cards render unchanged.
- **frontend api/types.ts** — Add RevisionVerdict + RevisionItem types; extend CraftReadResponse with revision_verdict?; extend UploadCard with prior_video_id? and revision_summary?.

## Risks

- **False 'Fixed' from read non-determinism — the new read randomly omits the prior change_type even though the issue persists (the documented core risk; re-uploading the SAME reel yields different reads).**
  - Absence of the tag is NECESSARY but NOT SUFFICIENT. Fixed requires the independent frames-grounded verifier to return resolved+high. The verifier never sees the new read text, so read variance cannot drive it. When the gates disagree (tag gone but verifier still_there), we trust the verifier and say 'Still there'.
- **Verifier hallucinates 'resolved' on noisy frames (vlm_perception is documented as noisy/biased), emitting a false success.**
  - Require confidence='high' for Fixed; medium degrades to 'Looks better — check yourself'; cant_tell suppresses. Every verdict is tap-to-seek auditable (TimeChip), so a wrong call lands the creator on the actual frame and self-defuses — the same trust affordance the read already uses. Conservative default: when unsure, never claim fixed.
- **Prior reel deleted (DELETE /me/uploads exists) or its corpus VideoFeatures wiped on redeploy, leaving the verdict unable to read the prior issue.**
  - Snapshot prior_issue_text/timestamp/change_type INTO revision_verdict at compute time; the verdict is self-contained and never re-reads the prior row. Verdict computes once at upload, persisted. If the prior Upload is gone at link time, fall back to a plain read (no verdict) — never a fabricated one.
- **Wrong link — creator re-checks against an unrelated reel, so 'fixed/still-there' compares apples to oranges.**
  - Linking is explicit and creator-asserted (they tap re-check on a specific read), not auto-inferred — the creator owns the intent. The verifier is anchored to the SPECIFIC prior issue text, so an unrelated new reel simply returns still_there/cant_tell (the issue isn't meaningfully present), not a false fixed.
- **Added latency/cost on re-upload (one extra VLM frame call per prior blind_spot, up to 4).**
  - Verifier runs only when prior_video_id is set (re-checks are a fraction of uploads); ~10-20s and ~$0.01-0.04, hidden inside the existing 'building your read' takeover. Fully async/best-effort: a verifier timeout logs and the verdict line degrades to 'can't verify', never blocking the read.
- **Provider not configured (craft_read_base_url unset) — verify_fix_addressed requires the openai/Qwen-VL endpoint, like the grounding gate.**
  - verify_fix_addressed returns None without a provider; compare_revision then emits only the cheap absence-gate result as 'looks_better/can't-confirm' wording (never 'fixed'), and the read still renders fully. No hard dependency added beyond what ground_and_gate already needs.
